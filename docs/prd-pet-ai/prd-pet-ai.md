## 1. 背景与目标

### 1.1 项目背景

中国宠物市场 2024 年规模超过 3400 亿元，其中宠物保健品/处方粮增速超过 700%，电商渠道占比持续提升。然而现有产品存在严重的体验割裂问题：宠物主需要在多个 App 之间切换（健康咨询→保健品选购→下单），AI 辅助健康诊断与电商购物的一体化闭环在国内尚无头部竞品。

本产品旨在打造一个集「AI 症状初诊 + 个性化保健品推荐 + 宠物用品电商」于一体的移动 App，以 AI 驱动的健康建议为核心差异化，形成「诊断即推荐，推荐即购买」的闭环体验。

> 立项验证详情见附录 A。

### 1.2 业务目标

1. **上线 6 个月内**：MAU 达到 10 万，AI 初诊使用率 ≥60%，初诊→电商转化率 ≥15%
2. **上线 12 个月内**：月 GMV 突破 500 万元，30 天留存率 ≥40%
3. **商业可持续**：通过电商差价（毛利 30-40%）和订阅服务（¥29-69/月）覆盖运营成本

### 1.3 成功指标

| 指标 | 目标值 | 衡量方式 |
|------|--------|----------|
| MAU | ≥10 万（6个月） | 应用统计后台 |
| AI 初诊次数/月 | ≥20 万次 | 服务端日志 |
| 初诊→电商转化率 | ≥15% | 漏斗分析 |
| 月 GMV | ≥500 万元（12个月） | 订单系统 |
| 30 天留存率 | ≥40% | 用户行为分析 |
| 平均会话时长 | ≥5 分钟 | 埋点统计 |
| AI 推荐满意度 | ≥4.0/5.0 | 用户评分 |

## 2. 用户画像

| 角色 | 特征 | 核心需求 | 使用场景 |
|------|------|----------|----------|
| **都市宠物主（主要）** | 25-35 岁，一线/新一线城市，月收入 8000+，养猫/狗 1-3 年，时间紧张 | 快速判断宠物是否需要就医；买到适合自家宠物的保健品 | 发现宠物异常时立即打开 App 问诊；日常购买宠物用品时 |
| **新手宠物主** | 养宠不足 1 年，缺乏经验，对宠物健康知识匮乏 | 入门健康指导；避免买错产品 | 宠物出现任何不适时；为宠物选购第一批保健品时 |
| **多宠家庭** | 同时养 2+ 只宠物，对管理效率要求高 | 同时管理多只宠物健康档案；批量购买用品 | 统一查看所有宠物健康状态；节日批量购物时 |

**核心闭环流程（P0）：**

```
注册/登录 → 创建宠物档案 → 描述症状（AI初诊）→ 获得健康评估
    → 查看推荐保健品 → 加入购物车 → 结算支付 → 订单跟踪
```

**日常使用流程：**

```
打开 App → 首页（宠物档案概览）→ 选择宠物 → AI 初诊 / 直接浏览商品
```

### 6.3 异常场景处理

| 异常场景 | 用户表现 | 处理方式 |
|----------|----------|----------|
| AI 服务超时 | 10 秒无响应 | 显示"服务繁忙，请稍后重试"，保留用户输入内容 |
| 网络断开 | 操作无响应 | Toast 提示"网络连接失败，请检查网络"；支持重新加载 |
| 宠物档案为空 | 进入 AI 初诊 | 引导先创建宠物档案；也支持临时问诊（不绑定档案） |
| 商品库存不足 | 加购时 | 显示"已售罄"或"库存不足 X 件"，提供到货提醒 |
| 支付失败 | 支付页面 | 显示失败原因，提供重试入口，订单保留 30 分钟 |

## 7. 技术方案概要

### 7.1 技术选型

| 层级 | 技术选型 | 说明 |
|------|----------|------|
| 移动端 | React Native | 跨平台（iOS/Android）开发，降低维护成本 |
| 后端 | Node.js + TypeScript | API 服务，适合高并发 I/O 场景 |
| AI 引擎 | OpenAI GPT-4o API + 症状决策树 | MVP 阶段采用 LLM + 规则混合方案，避免自训练模型成本 |
| 数据库 | PostgreSQL（关系数据）+ Redis（缓存）| 用户/订单/档案结构化存储；热点数据缓存 |
| 文件存储 | 阿里云 OSS + CDN | 宠物图片存储与加速 |
| 支付 | 微信支付 + 支付宝 SDK | 覆盖主流支付方式 |
| 部署 | Docker + 阿里云 ECS / K8s | 容器化部署，按需扩缩容 |

### 7.2 系统架构概述

系统采用前后端分离架构，后端按业务域拆分为以下服务：
- **用户服务**：认证、授权、用户信息管理
- **宠物服务**：宠物档案 CRUD、健康记录
- **AI 诊断服务**：对接 LLM API，管理会话上下文和症状决策树
- **推荐服务**：基于初诊结果和宠物档案计算推荐商品
- **电商服务**：商品目录、购物车、订单、支付网关对接

> 详细架构方案待 `architect` Skill 完成后补充，参见 `architecture-pet-ai.md`（待生成）。

### 7.3 关键接口定义

| 接口 | 方法 | 路径 | 描述 |
|------|------|------|------|
| AI 初诊 | POST | `/api/v1/diagnosis` | 提交症状，返回 AI 健康评估 |
| 获取推荐 | GET | `/api/v1/recommendations` | 基于初诊ID获取推荐商品列表 |
| 宠物档案列表 | GET | `/api/v1/pets` | 获取用户宠物列表 |
| 创建宠物档案 | POST | `/api/v1/pets` | 创建新宠物档案 |
| 商品搜索 | GET | `/api/v1/products` | 关键词/分类搜索商品 |
| 创建订单 | POST | `/api/v1/orders` | 创建订单并发起支付 |
| 用户登录 | POST | `/api/v1/auth/login` | 手机号+验证码登录 |

## 9. 风险与依赖

### 9.1 风险项

| 风险 | 概率 | 影响 | 应对措施 |
|------|------|------|----------|
| AI 健康建议合规风险 | 高 | 高 | 法律顾问咨询；所有 AI 输出添加"仅供参考，不构成医疗建议"免责声明；严格限定健康建议而非医疗诊断 |
| LLM API 成本超预期 | 中 | 中 | 设置每日 API 调用上限；对重复性问题建立缓存机制；后期考虑微调小模型替代 |
| 供应链冷启动困难 | 中 | 高 | 初期与 3-5 家知名保健品品牌合作，采用代发货模式，无需自建仓储 |
| 用户信任度不足 | 高 | 高 | 邀请执业兽医顾问团队参与内容审核；展示兽医背书；强调"辅助工具"定位 |
| iOS 审核风险 | 中 | 高 | 提前与苹果沟通医疗健康类 App 审核要求；避免使用"诊断"等医疗术语 |

## 附录 A：需求分析报告

### A.1 需求概述

**产品定位**：宠物 AI 初诊 + 保健品建议 + 宠物用品销售一体化 App

**核心功能点（5个）**：
1. **AI 症状初诊**：拍照/文字问答，给出健康评估与就医建议
2. **个性化保健品推荐**：基于初诊结果 + 宠物档案（品种/年龄/体重）智能推荐
3. **宠物用品电商**：完整购物车、支付、订单、售后流程
4. **健康档案管理**：体重趋势、疫苗记录、就诊历史
5. **在线兽医咨询（可选）**：AI 初诊后一键转接兽医（P2 优先级）

**目标用户**：⚠️ 假设性画像 — 25–35 岁都市白领，养猫/狗 1–3 年，月收入 8000+，关注宠物健康但无时间频繁就医。

**核心痛点**：
- 不知宠物是否需要就医，盲目就医成本高（单次 200–2000 元）
- 保健品市场信息不对称，买错产品既浪费钱又可能危害健康
- 从诊断到购买产品全程体验割裂，需在多个 App 间切换

### A.3 竞品分析

**国内主要竞品**：
