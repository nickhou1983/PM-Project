## 1. 背景与目标

### 1.1 项目背景

中国宠物市场 2024 年规模超过 3400 亿元，其中宠物保健品/处方粮增速超过 700%，电商渠道占比持续提升。然而现有产品存在严重的体验割裂问题：宠物主需要在多个 App 之间切换（健康咨询→保健品选购→下单），AI 辅助健康诊断与电商购物的一体化闭环在国内尚无头部竞品。

在此基础上，带宠出行场景正在快速增长，但宠物主仍需在地图、点评、社群和短视频平台之间反复搜索，才能判断某个地点是否真正宠物友好。场所规则不透明、数据分散、难以结合自家宠物特征筛选，是当前最大的体验断层。

本产品现阶段在原有「AI 症状初诊 + 个性化保健品推荐 + 宠物用品电商」闭环上，新增「宠物友好场景发现」模块，形成「健康管理 + 出行场景发现 + 电商转化」的一体化宠物生活服务平台。

> 立项验证详情见附录 A。

### 1.2 业务目标

1. **上线 6 个月内**：MAU 达到 10 万，AI 初诊使用率 ≥60%，地图场景发现功能月使用率 ≥35%，初诊→电商转化率 ≥15%
2. **上线 9 个月内**：完成 3 个重点城市（上海/北京/广州）宠物友好场所覆盖，每城有效 POI ≥500 个，场所详情页点击→导航转化率 ≥20%
3. **上线 12 个月内**：月 GMV 突破 500 万元，30 天留存率 ≥40%，商家合作场所数 ≥300 家
4. **商业可持续**：通过电商差价（毛利 30-40%）、订阅服务（¥29-69/月）与商家推广位收入覆盖运营成本

### 1.3 成功指标

| 指标 | 目标值 | 衡量方式 |
|------|--------|----------|
| MAU | ≥10 万（6个月） | 应用统计后台 |

| 地图场景发现 DAU | ≥3.5 万（6个月） | 地图埋点 |

## 2. 用户画像

| 角色 | 特征 | 核心需求 | 使用场景 |
|------|------|----------|----------|
| **都市宠物主（主要）** | 25-35 岁，一线/新一线城市，月收入 8000+，养猫/狗 1-3 年，时间紧张且愿意带宠外出 | 快速判断宠物是否需要就医；买到适合自家宠物的保健品；找到真正允许携宠进入的场所 | 发现宠物异常时立即打开 App 问诊；周末带宠出行前查询附近友好场所 |
| **新手宠物主** | 养宠不足 1 年，缺乏经验，对宠物健康知识和出行规则都不熟悉 | 入门健康指导；避免买错产品；避免因不了解场所规则导致白跑 | 宠物出现任何不适时；首次带宠去商场/咖啡馆/公园前 |
| **多宠家庭** | 同时养 2+ 只宠物，对管理效率要求高，带宠出行时更关注场地限制 | 同时管理多只宠物健康档案；快速判断不同体型/品种宠物是否适合同一场所 | 统一查看所有宠物健康状态；规划家庭周末带宠路线 |
| **宠物友好商家运营者（次要）** | 咖啡馆/餐厅/公园/酒店运营人员，希望吸引高质量宠物客群 | 低成本展示自己的宠物友好规则与服务特色；获得真实用户反馈 | 入驻宠物友好地图，维护营业信息和入宠规则 |

## 6. 交互设计

### 6.1 核心用户流程

**核心闭环流程（P0）：**

```
注册/登录 → 创建宠物档案 → 描述症状（AI初诊）→ 获得健康评估
    → 查看推荐保健品 → 加入购物车 → 结算支付 → 订单跟踪
```

**带宠出行流程（P0/P1）：**

```
打开 App → 选择当前宠物 → 查看附近宠物友好地图 → 筛选场所
    → 查看场所详情与入宠规则 → 跳转导航前往 → 到店后打卡评价
```

### 6.3 异常场景处理

| 异常场景 | 用户表现 | 处理方式 |
|----------|----------|----------|
| AI 服务超时 | 10 秒无响应 | 显示"服务繁忙，请稍后重试"，保留用户输入内容 |
| 网络断开 | 操作无响应 | Toast 提示"网络连接失败，请检查网络"；支持重新加载 |
| 宠物档案为空 | 进入 AI 初诊或场景发现 | 引导先创建宠物档案；也支持临时问诊/游客查看基础场所列表 |

| 支付失败 | 支付页面 | 显示失败原因，提供重试入口，订单保留 30 分钟 |
| 定位权限被拒绝 | 首次打开地图页 | 说明需要定位以推荐附近场所，并允许手动切换城市 |
| 场所规则缺失 | 查看场所详情 | 标记"规则待确认"并提示用户以电话/评价补充为准，避免误导 |

## 7. 技术方案概要

### 7.1 技术选型

| 层级 | 技术选型 | 说明 |
|------|----------|------|
| 移动端 | React Native | 跨平台（iOS/Android）开发，降低维护成本 |
| 后端 | Node.js + TypeScript | API 服务，适合高并发 I/O 场景 |
| AI 引擎 | OpenAI GPT-4o API + 症状决策树 | MVP 阶段采用 LLM + 规则混合方案，避免自训练模型成本 |
| 地图与定位 | 高德地图 SDK（候选：腾讯地图 SDK） | 国内地图能力成熟，支持定位、POI 检索、导航跳转 |
| 数据库 | PostgreSQL（关系数据）+ Redis（缓存）| 用户/订单/档案结构化存储；热点数据缓存 |
| 地理数据 | PostgreSQL + PostGIS（待 Architect 确认） | 支撑附近场所检索、空间筛选与聚合 |
| 文件存储 | 阿里云 OSS + CDN | 宠物图片、打卡图片存储与加速 |
| 支付 | 微信支付 + 支付宝 SDK | 覆盖主流支付方式 |
| 部署 | Docker + 阿里云 ECS / K8s | 容器化部署，按需扩缩容 |

### 7.2 系统架构概述

系统采用前后端分离架构，后端按业务域拆分为以下服务：
- **用户服务**：认证、授权、用户信息管理
- **宠物服务**：宠物档案 CRUD、健康记录
- **场景发现服务**：宠物友好场所 POI、地图筛选、规则标签、评价聚合
- **AI 诊断服务**：对接 LLM API，管理会话上下文和症状决策树
- **推荐服务**：基于初诊结果和宠物档案计算推荐商品
- **电商服务**：商品目录、购物车、订单、支付网关对接

## 10. 术语表

| 术语 | 定义 |
|------|------|
| AI 初诊 | 用户描述宠物症状后，AI 给出健康状态评估和建议，非医学诊断行为，不具有法律效力 |
| 健康评估分级 | AI 对宠物健康状态的三级分类：🟢 正常（可正常观察）、🟡 关注（密切观察，必要时就医）、🔴 就医（建议尽快就医） |
| 宠物档案 | 包含宠物基本信息（品种、年龄、体重、性别、绝育状态）和健康记录（体重趋势、疫苗记录、就诊历史）的数据集合 |
| POI | Point of Interest，地图中的兴趣点数据，本产品中指宠物友好场所 |
| 宠物友好规则 | 场所允许携宠进入所附带的限制与要求，如牵引、体型、疫苗证明、室内外区域限制 |
| LBS | Location Based Service，基于位置提供附近场所、导航、距离等能力的服务 |
| 初诊→推荐转化 | 用户完成 AI 初诊后点击查看并购买推荐商品的行为，是核心业务转化路径 |
| RICE 评分 | 功能优先级评估框架，RICE = (Reach × Impact × Confidence) / Effort |
| GMV | 商品交易总额（Gross Merchandise Volume），电商业务核心规模指标 |
| 代发货 | 商家在收到订单后，由供应商直接发货给消费者，平台不需要持有库存 |
| LLM | 大语言模型（Large Language Model），用于实现 AI 症状问答和健康建议生成 |

## 附录 A：需求分析报告

### A.1 需求概述

**增量需求**：开发宠物友好场景查询能力，支持地图检索、场所详情、入宠规则、导航与打卡评价，并与现有宠物档案联动。

**分析模式**：🔄 迭代分析（基于 `prd-pet-ai.md` v1.0.0）

**核心功能点（5个）**：
1. **附近宠物友好地图**：按当前位置展示周边咖啡馆、商场、公园、酒店等场所
2. **场所详情与入宠规则**：结构化展示营业时间、牵引要求、体型限制、配套设施
3. **条件筛选与路线导航**：按宠物类型、体型、场景偏好筛选，并跳转地图导航
4. **打卡评价**：用户发布到店体验、图片和标签，为后续用户提供参考
5. **宠物档案联动匹配**：结合宠物档案提示场所与当前宠物是否匹配

**目标用户**：⚠️ 假设性画像 — 25–35 岁都市宠物主，周末有明显带宠出行需求，常在地图、社交平台和点评平台之间切换搜集信息。

**核心痛点**：
- 宠物友好场所信息分散，用户需跨平台反复确认，效率低
- 场所规则不透明，常出现到店后才发现不能进或限制较多的情况
- 现有地图平台不理解用户当前宠物特征，无法做更细粒度的场所匹配

**查询状态**：⚠️ 未查询（当前执行环境缺少 `FEISHU_APP_ID` / `FEISHU_APP_SECRET`，未完成飞书 OAuth）

**本地文档比对结果**：现有 `prd-pet-ai` 基线中无地图/LBS/场景发现模块，仅有 `pet-profile`、`ai-diagnosis`、`health-recommendation`、`ecommerce` 等能力；本次增量需求属于横向扩展。

**重复判定**：🆕 全新需求（针对现有产品的新模块）

### A.3 竞品分析

**市场空白**：国内缺乏一款将宠物档案、场所规则与地图场景发现深度整合的垂直宠物产品。
