### 市场洞察

- **市场空白**：当前市场形成两极分化——
  - 一极是**通用但浅层**的提示词工具（Promptsera、FlowGPT），覆盖多模型但不理解平台专属语法
  - 另一极是**深度但单一**的工具（SoPrompts 仅 Sora），或平台内置编辑器（即梦自带、MJ Discord bot）
  - **"宽且深"的提示词编辑器——同时覆盖图像+视频多模型、且深度适配每个模型格式规范的工具，目前市场上不存在**
- **差异化机会**：
  1. **"一次编写，N 模型输出"**：用户在统一编辑器中创作，一键输出适配 MJ/SD/Sora/Runway/Kling 等各模型的格式化提示词——这是核心差异化壁垒
  2. **结构化参数 + AI 双驱动 + 模型专属适配**：不同于 Promptsera 等浅层工具，深度理解每个模型的参数语法（MJ 的 `--ar 16:9 --v 7`、SD 的 `(keyword:1.3)`、Sora 的自然语言运镜指令等）
  3. **模型智能推荐**：根据用户的创作意图，推荐最适合的模型——如"产品照"推荐 DALL-E 3.5（精准）、"概念艺术"推荐 MJ v7（美感）、"电影感短片"推荐 Runway Gen-4.5
  4. **图像+视频双赛道覆盖**：2026 年"图→视频"工作流（先 AI 生成图像再转视频）占 32.6% 订单量（Vivideo 数据），天然需要图像+视频提示词工具链
- **潜在风险**：
  1. 模型格式更新频率高——MJ v7 和 v6 参数语法已有差异，维护成本随模型数量线性增长
  2. AI 模型平台自身正在增强提示词辅助能力（如 DALL-E 3.5 的对话式迭代、Runway 的 Gen-4.5 智能提示）
  3. 多模型聚合器（PixelBunny、Tingu）可能向提示词辅助功能扩展
  4. 市场碎片化——新模型不断涌现（2026 年 Veo 3.1 已获 96.4% 份额超越 Sora），追赶所有模型的成本高

### SWOT 分析

| | 有利 | 不利 |
|---|------|------|
| **内部** | **S 优势**：已有 Sora+Seedance 双模型适配基础架构，扩展模型覆盖有复用性；结构化参数面板+AI 扩写能力可复用；"宽且深"的定位在市场有差异化空间 | **W 劣势**：模型覆盖面扩大 = 维护成本倍增（每个模型需独立的格式规则、参数映射、最佳实践维护）；团队规模小可能无法快速覆盖所有模型；图像模型参数体系与视频模型差异大，需新增面板 |
| **外部** | **O 机会**：AI 视频市场 2025→2034 年 CAGR 46%（$1.23B→$21.61B）；生成式 AI 市场 2026→2035 年 CAGR 37%（$55.5B→$1206B）；图→视频工作流趋势（32.6% 订单量），天然需要图像+视频提示词工具链；提示词工程工具被列为 2026 年 SaaS 增长快品类 | **T 威胁**：Veo 3.1 已占 96.4% 视频生成市场份额，Sora 仅 2%——模型格局变化剧烈；平台方内置提示词辅助越来越强；模型格式更新频率高增加维护负担；竞品可快速跟进多模型适配 |

## 5. 可落地性快评

### UI 复杂度初评

| 评估项 | 结论 | 说明 |
|--------|------|------|
| **核心用户路径** | 一般 | 在原有"输入→AI优化→选模型→复制"基础上，需新增：模型选择/切换（10+ 模型）、图像参数面板（与视频参数差异大）、模型对比视图、模型推荐。流程从 3 步变为 4-5 步 |
| **页面与状态复杂度** | 中→高 | 1. 参数面板需根据模型类型（图像 vs 视频）动态切换，图像模型需新增：宽高比参数（MJ `--ar`）、风格版本（`--v 7`/`--niji`）、负向提示词（SD negative prompt）、权重语法等<br>2. 模型选择器需处理 10+ 模型的分类、状态、参数差异<br>3. 预估核心页面从 6 增至 8-9 个 |
| **交互风险** | 中 | 1. 多模型参数面板的动态切换交互（不同模型参数维度不同）<br>2. 模型对比视图需展示 N 个模型的格式差异（N 从 2 扩到 5+）<br>3. 图像 vs 视频模式切换的 UX 一致性 |
| **MVP 可收敛性** | 中 | 不建议一次覆盖所有模型。MVP 可收敛为：在现有 Sora+Seedance 基础上，先扩展 3 个高需求模型（建议优先 Midjourney + Runway + Kling），后续迭代再逐步覆盖 |

**UI 复杂度结论**：**中偏高**

> 建议：MVP 阶段不追求全模型覆盖，先扩展至 5 个模型（Sora + Seedance + Midjourney + Runway + Kling），验证"多模型适配"价值后再扩展。

### 技术可行性初评

| 评估项 | 结论 | 说明 |
|--------|------|------|
| **技术可行性** | 高 | 核心技术挑战是"模型格式适配引擎"的设计——将统一参数映射为各模型专属格式。这属于规则引擎/模板引擎范畴，技术成熟无阻塞。不涉及推荐算法、实时协同等高难点 |
| **实现复杂度** | 中 | 1. 需设计**模型配置层**（每个模型的参数 schema、格式模板、最佳实践规则）——可配置化存储为 JSON/DB<br>2. 图像模型参数体系与视频模型不同，需新增参数面板组件<br>3. MJ 的 `--参数` 语法、SD 的 `(keyword:weight)` 语法、DALL-E 的自然语言，格式差异需逐一实现<br>4. 预估比 v1.0 增加 40-60% 工作量 |
| **关键依赖** | 中 | 1. 各模型提示词格式文档（Sora / MJ / SD / Runway / Kling 等，大多公开可用）<br>2. OpenAI API（AI 扩写，已有）<br>3. 模型格式更新的持续追踪维护（运营性成本） |
| **系统风险** | 低→中 | 1. 格式规则维护风险：模型更新频率高（MJ v6→v7 参数变更），需建立格式规则版本管理机制<br>2. 性能风险低：仍是标准 CRUD + API 调用<br>3. 需考虑模型配置的热更新能力（新增/修改模型不需要发版） |

**技术快评结论**：**建议缩范围推进**

> 建议分两阶段实施：
> - **Phase 1**（v2.0 MVP）：在现有架构上新增 3 个模型（Midjourney v7 + Runway Gen-4.5 + Kling 3.0），重构提示词引擎为可配置化架构
> - **Phase 2**（v2.1）：扩展至文生图模型（DALL-E 3.5 + Stable Diffusion/Flux 2 + Adobe Firefly 3），新增图像参数面板

## 7. 结论与建议

**结论**：⚠️ 建议缩范围推进

**理由**：
通用模型提示词生成是高价值方向——市场需求强（AI 生成式市场高速增长 + 提示词工具是快增品类）、竞争定位清晰（"宽且深"的差异化）、商业逻辑通顺（多模型输出是天然付费转化点）。但一次性覆盖 10+ 模型的 UI 复杂度和格式维护成本过高，建议**分阶段扩展模型覆盖**，先用 MVP 验证"多模型适配"的核心价值。

**下一步行动**：

1. **Phase 1（v2.0 MVP，建议 2 周）**：在现有 Sora + Seedance 基础上，新增 **Midjourney v7 + Runway Gen-4.5 + Kling 3.0** 共 5 个模型适配；重构提示词引擎为可配置化架构（模型格式规则存储为 JSON 配置，支持热更新）
2. **Phase 2（v2.1，建议 2 周）**：扩展至文生图模型（DALL-E 3.5 + Stable Diffusion/Flux 2 + Adobe Firefly 3），新增图像专属参数面板（负向提示词、权重语法、风格版本等）；新增模型智能推荐功能
3. **数据验证**：MVP 上线后追踪"模型切换次数/用户"和"各模型使用分布"，验证多模型适配的真实需求强度，数据驱动后续模型优先级
4. **格式维护机制**：建立模型格式规则版本管理和社区贡献机制（如开源格式配置文件），降低长期维护成本

> 💡 **如果结论为「⚠️ 建议缩范围推进」**，先明确 Phase 1 MVP 范围（5 模型 + 引擎重构），再使用 `requirement-doc` Skill 更新 PRD。

> 💡 **评审闭环**：PRD 更新后，建议使用 `gate_review` Agent 做正式评审（Go/No-Go 决策），确保质量达标后推进架构设计。

> 💡 **原型展示**：PRD 生成后，可使用 `modao-prototype` Skill 将核心页面导入墨刀，用于产品评审和团队协作展示。
