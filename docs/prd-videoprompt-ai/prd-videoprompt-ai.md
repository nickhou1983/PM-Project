## 1. 背景与目标

### 1.1 项目背景

2026 年，AI 生成式内容市场进入爆发期。视频生成领域，Sora 2、Seedance 2.0、Runway Gen-4.5、Kling 3.0、Veo 3.1、Pika 等模型百花齐放；图像生成领域，Midjourney v7、DALL-E 3.5、Stable Diffusion/Flux 2、Adobe Firefly 3 等工具被大量创作者日常使用。AI 视频生成市场规模 2026 年预计达 $18.1 亿，CAGR 46%；生成式 AI 整体市场预计 $555 亿，CAGR 37%。

然而，不同 AI 模型的提示词格式和最佳实践差异极大：Midjourney 使用 `--ar` `--v` 参数语法，Stable Diffusion 使用 `(keyword:1.3)` 权重语法，Sora 偏好结构化自然语言运镜指令，Seedance 支持 @ 引用语法。用户在多个模型间切换的学习成本极高，且市场缺乏"跨模型深度适配"的统一提示词工具。

现有竞品呈两极分化：一类是"宽但浅"的通用工具（如 Promptsera、FlowGPT），覆盖多模型但不理解平台专属语法；另一类是"深但窄"的垂直工具（如 SoPrompts 仅支持 Sora）。**"宽且深"——同时覆盖视频+图像多模型、且深度适配每个模型格式规范的智能提示词编辑器，目前市场上不存在。**

> 📝 v1.0 立项验证详情见 [analysis-report.md](analysis-report.md)。v2.0 通用模型扩展分析见 [analysis-report-universal-model.md](analysis-report-universal-model.md)。

### 1.2 业务目标

1. **降低 AI 视频/图像提示词创作门槛**：让不懂专业术语的用户也能写出高质量提示词
2. **打造「一次编写，全模型输出」的第一工具**：覆盖主流视频模型（Sora、Seedance、Runway、Kling、Veo、Pika）+ 图像模型（Midjourney、DALL-E、SD/Flux、Firefly），抢占"宽且深"的市场空白
3. **构建提示词工具型 SaaS 产品**：通过 Freemium 模式实现商业化（免费限 2 模型，Pro 解锁全部）
4. **分阶段验证市场需求**：Phase 1（v2.0 MVP）扩展至 5 个视频模型验证多模型价值，Phase 2（v2.1）覆盖图像模型

### 4.2 功能详细描述

#### 4.2.1 结构化参数面板

- **所属模块**：提示词编辑器
- **描述**：提供可视化的参数选择面板，用户通过下拉菜单、标签选择等方式设置视频提示词的各维度参数，系统实时组合为完整提示词文本
- **参数维度**：
  - **主体描述**：文本输入（如"一个穿红色连衣裙的女孩"）
  - **动作**：文本输入（如"在雨中奔跑"）
  - **环境/场景**：下拉 + 文本（城市街道/森林/海边/室内/自定义）
  - **运镜方式**：下拉选择（推轨/摇臂/航拍/手持/静态/环绕/跟随 等，附中文说明）
  - **光影**：下拉选择（自然光/黄金时段/伦勃朗光/逆光/霓虹灯/烛光 等）
  - **情绪/氛围**：标签选择（欢快/忧伤/紧张/宁静/史诗/神秘 等）
  - **视频风格**：下拉选择（电影感/纪录片/MV/广告/动画/赛博朋克 等）
  - **时长**：滑块（4s / 8s / 10s / 16s）
  - **画面比例**：选择（16:9 / 9:16 / 1:1）
- **输入**：用户交互选择/输入各参数
- **处理逻辑**：根据选中参数，按「主体 + 动作 + 环境 + 运镜 + 光影 + 情绪 + 风格」结构拼接提示词
- **输出**：实时更新的提示词文本预览
- **异常场景**：必填项（主体描述）为空时，提示用户填写

#### 4.2.2 AI 一键优化

- **所属模块**：AI 智能扩写
- **描述**：用户输入简单的场景描述或已拼接的粗糙提示词，AI 将其扩写为包含丰富视觉细节、运镜指令和光影描述的专业提示词
- **输入**：用户输入的文本（≥ 5 个字符）
- **处理逻辑**：
  1. 调用 LLM API（OpenAI/Claude），传入用户文本 + 目标模型（Sora/Seedance）+ 系统 Prompt
  2. 系统 Prompt 包含视频提示词最佳实践规则（基于 OpenAI Sora Prompting Guide 和 Seedance 官方手册）
  3. 返回扩写后的完整提示词
- **输出**：AI 生成的优化提示词，用户可直接采用或继续编辑
- **异常场景**：
  - 输入过短（< 5 字符）：提示"请输入更详细的描述"
  - API 调用失败：显示错误提示并支持重试
  - API 响应超时（> 10s）：显示加载动画并提示"生成中，请稍候"

#### 4.2.3 Sora 格式输出

- **所属模块**：模型适配
- **描述**：将编辑器中的提示词按 Sora 2 的最佳实践格式化输出
- **格式规则**（基于 OpenAI 官方 Prompting Guide）：
  - 使用英文输出（自动翻译）
  - 按「画面主体 + 具体动作 + 环境细节 + 运镜指令 + 光影描述 + 画面风格」结构
  - 动作描述具体化（避免 "moves quickly"，改用 "jogs three steps and stops"）
  - 环境描述使用可见的物理细节（避免 "beautiful street"，改用 "wet asphalt, neon signs reflecting in puddles"）
  - 技术参数放在最后（如 "anamorphic 2.0x lens, shallow DOF"）
- **输出**：格式化的 Sora 提示词文本

#### 4.2.4 Seedance 格式输出

- **所属模块**：模型适配
- **描述**：将编辑器中的提示词按 Seedance 2.0 的格式和最佳实践输出
- **格式规则**（基于即梦官方手册）：
  - 支持中文输出（Seedance 对中文支持良好）
  - 支持 @ 引用语法标注（如 @视频1、@图片1）
  - 运镜和转场描述符合 Seedance 的理解模式
  - 可附加参考视频/图片的引用说明
- **输出**：格式化的 Seedance 提示词文本

#### 4.2.5 Runway Gen-4.5 格式输出（v2.0 新增）

- **所属模块**：模型适配
- **描述**：将编辑器中的提示词按 Runway Gen-4.5 的最佳实践格式化输出
- **格式规则**：
  - 英文输出
  - 强调 camera direction 关键词（pan left, zoom in, tracking shot 等）
  - compositional consistency 描述（保持人物/场景一致性的约束指令）
  - 支持运动强度（motion intensity）参数标注
- **输出**：格式化的 Runway 提示词文本

#### 4.2.6 Kling 3.0 格式输出（v2.0 新增）

- **所属模块**：模型适配
- **描述**：将编辑器中的提示词按 Kling/可灵 3.0 的格式和最佳实践输出
- **格式规则**：
  - 支持中英双语输出
  - 角色一致性指令（保持角色外观、服装、表情的一致性描述）
  - 明确标注帧率和时长参数
  - 镜头运动描述适配可灵的理解模式
- **输出**：格式化的 Kling 提示词文本

#### 4.2.7 模型配置层（v2.0 新增）

- **所属模块**：模型适配引擎
- **描述**：将每个模型的格式规则、参数 schema、最佳实践规则存储为 JSON 配置文件，实现模型规则与代码解耦，支持热更新（新增/修改模型时无需发版）
- **配置 schema 包含**：
  - `modelId`：模型唯一标识
  - `name`：模型显示名称
  - `type`：模型类型（video / image）
  - `language`：推荐输出语言（en / zh / both）
  - `parameterSchema`：该模型支持的参数维度和选项
  - `formatTemplate`：提示词拼接模板
  - `syntaxRules`：专属语法规则（如 MJ 的 `--` 参数、SD 的权重语法）
  - `bestPractices`：最佳实践提示（用于 AI 扩写时的 system prompt）
  - `version`：格式规则版本号
- **输入**：JSON 配置文件（存储于数据库或静态文件）
- **输出**：动态加载的模型格式规则，驱动参数面板和格式输出
- **异常场景**：配置文件解析失败时回退到上一个有效版本

#### 4.2.8 统一参数抽象层（v2.0 新增）

- **所属模块**：模型适配引擎
- **描述**：定义一套模型无关的通用参数体系，用户在编辑器中操作通用参数，系统按目标模型的配置自动映射为专属格式
- **通用参数映射示例**：

| 通用参数 | Sora 映射 | MJ 映射 | SD 映射 | Kling 映射 |
|----------|-----------|---------|---------|------------|
| 画面比例 16:9 | "aspect ratio: 16:9" | `--ar 16:9` | 分辨率 1920x1080 | "16:9 画面比例" |
| 风格：电影感 | "cinematic style" | `--style cinematic` | "(cinematic:1.3)" | "电影感风格" |
| 运镜：推轨 | "dolly tracking shot" | N/A（图像无运镜） | N/A | "推轨镜头" |

- **处理逻辑**：读取目标模型配置 → 遍历用户选择的通用参数 → 按映射规则转换 → 按模板拼接输出
- **异常场景**：目标模型不支持某通用参数时（如图像模型无运镜），自动忽略并在 UI 上灰显该参数

#### 4.2.9 Midjourney v7 格式输出（v2.1 新增）

- **所属模块**：模型适配
- **描述**：将编辑器中的提示词按 Midjourney v7 的参数语法格式化输出
- **格式规则**：
  - 英文视觉形容词驱动的描述风格（lean into visual adjectives + composition）
  - 参数以 `--` 开头追加在描述文本之后
  - 支持的参数：`--ar`（画面比例）、`--v 7`（版本）、`--style`（风格）、`--chaos`（混乱度）、`--stylize`（风格化强度）、`--niji`（二次元模式）
  - 支持负向提示词语法 `--no {keyword}`
- **输出示例**：`A young woman in white linen shirt spinning under cherry blossoms in Kyoto, golden hour, cinematic wide angle, soft bokeh --ar 16:9 --v 7 --style raw --stylize 300`

#### 4.2.10 SD/Flux 2 格式输出（v2.1 新增）

- **所属模块**：模型适配
- **描述**：将编辑器中的提示词按 Stable Diffusion / Flux 2 的权重语法格式化输出
- **格式规则**：
  - 英文关键词列表风格（逗号分隔）
  - 支持 `(keyword:weight)` 权重语法，权重范围 0.1-2.0
  - 支持独立的 negative prompt 区域
  - 关键词按重要性排序（主体 > 风格 > 环境 > 光影 > 技术参数）
- **输出示例**：
  - **Positive**：`(young woman in white linen shirt:1.2), spinning, (cherry blossoms:1.3), kyoto japan, ancient street, (golden hour lighting:1.1), cinematic, wide angle, shallow depth of field, bokeh, masterpiece, best quality`
  - **Negative**：`lowres, bad anatomy, bad hands, text, watermark, blurry, deformed`

## 7. 技术方案概要

### 7.1 技术选型

| 层级 | 技术选型 | 说明 |
|------|----------|------|
| 前端 | Next.js 14 + React 18 + TypeScript | SSR/SSG 支持 SEO，React 生态成熟，TypeScript 保障代码质量 |
| UI 框架 | Tailwind CSS + shadcn/ui | 高效开发 + 可定制组件库 |
| 后端 | Next.js API Routes (Serverless) | 前后端一体，MVP 快速交付，无需独立后端服务 |
| AI 服务 | OpenAI API (GPT-4o) | 用于 AI 扩写和提示词优化 |
| 数据库 | Supabase (PostgreSQL) | 免费额度充足，支持用户认证 + 数据存储 |
| 认证 | Supabase Auth (OAuth 2.0) | 支持 Google/GitHub 登录，与数据库无缝集成 |
| 部署 | Vercel | 与 Next.js 深度集成，全球 CDN，免费额度适合 MVP |
| 本地存储 | localStorage | 未登录用户的历史记录和偏好存储 |

### 7.2 系统架构概述

```
用户浏览器 (Next.js SSR)
    ├── 编辑器页面（客户端渲染，实时交互）
    │   ├── 通用参数面板（模型无关）
    │   ├── 模型专属参数面板（按目标模型动态加载）
    │   └── 模型选择器（视频模型 / 图像模型分类）
    ├── 模板库页面（SSG 静态生成，SEO 友好）
    └── 用户中心页面（需认证）

Next.js API Routes (Serverless Functions)
    ├── /api/ai/enhance     → 调用 OpenAI API 扩写提示词（传入目标模型 + 模型最佳实践）
    ├── /api/ai/translate    → 调用 OpenAI API 翻译提示词
    ├── /api/ai/recommend    → 根据创意描述推荐 Top 3 模型（v2.1）
    ├── /api/models          → 获取可用模型列表及配置（v2.0）
    ├── /api/models/:id      → 获取单个模型的格式规则配置（v2.0）
    ├── /api/prompts/*       → CRUD 用户提示词（Supabase）
    └── /api/auth/*          → 用户认证（Supabase Auth）

Supabase (PostgreSQL + Auth)
    ├── users 表
    ├── prompts 表（用户保存的提示词，新增 target_model 字段）
    ├── templates 表（预设模板数据，新增模型类型标签）
    └── model_configs 表（v2.0：模型格式规则 JSON 配置）
```

### 7.3 关键接口定义

| 接口 | 方法 | 路径 | 描述 |
|------|------|------|------|
| AI 扩写 | POST | /api/ai/enhance | 输入简单描述 + 目标模型，返回优化后的提示词（系统 prompt 按目标模型配置动态注入最佳实践） |
| AI 翻译 | POST | /api/ai/translate | 输入提示词 + 目标语言，返回翻译结果 |
| AI 模型推荐（v2.1） | POST | /api/ai/recommend | 输入创意描述，返回 Top 3 推荐模型 + 推荐理由 |
| 获取模型列表（v2.0） | GET | /api/models | 获取所有可用模型列表（含类型、名称、状态） |
| 获取模型配置（v2.0） | GET | /api/models/:id | 获取指定模型的格式规则配置（参数 schema + 格式模板 + 最佳实践） |
| 保存提示词 | POST | /api/prompts | 保存提示词到用户个人库（含目标模型标记） |
| 获取提示词列表 | GET | /api/prompts | 获取当前用户的提示词列表（分页，支持按模型筛选） |
| 删除提示词 | DELETE | /api/prompts/:id | 删除指定提示词 |
| 获取模板列表 | GET | /api/templates | 获取模板列表（支持分类 + 模型类型筛选） |

## 9. 风险与依赖

### 9.1 风险项

| 风险 | 概率 | 影响 | 应对措施 |
|------|------|------|----------|
| 多模型格式频繁变更 | 高 | 高 | 模型配置层采用 JSON 热更新，格式规则与代码解耦；建立各模型官方文档监控机制，版本变更 24h 内响应 |
| 模型数量增长导致维护负担 | 中 | 高 | 统一参数抽象层减少重复代码；每个模型格式引擎独立模块化，新增模型仅需配置 + 格式引擎两个文件 |
| LLM API 调用成本超预期 | 中 | 高 | MVP 阶段设置每日免费额度上限（5次/未登录用户，20次/免费用户）；使用 GPT-4o-mini 降低成本 |
| 平台自身内置类似功能 | 中 | 高 | 差异化定位为"跨模型一站式"工具，覆盖广度是单平台无法替代的核心优势；加速用户积累 |
| 覆盖广度稀释单模型深度 | 中 | 中 | 分阶段推进：v2.0 先做 5 个视频模型做深做透，v2.1 再扩展图像模型；每个模型必须通过格式准确度测试 |
| 图像模型参数复杂度高 | 中 | 中 | MJ 权重语法、SD 负向提示词等均通过专用参数面板降低门槛；复杂参数提供智能默认值和示例提示 |
| 用户付费意愿不足 | 中 | 中 | 前期免费运营收集数据，验证后再推进付费；探索 API 接入等 B 端变现路径 |
| 内容安全合规风险 | 低 | 高 | AI 输出经过内容安全过滤；参考平台内容政策制定输出规范 |

> **计算公式**：RICE 分 = (Reach × Impact × Confidence) / Effort

> - **Reach（影响面）**：一个迭代周期内，该功能直接影响的用户数量（1-10 等级）
> - **Impact（影响度）**：对单个用户的价值影响程度（3=巨大 / 2=高 / 1=中 / 0.5=低 / 0.25=极低）
> - **Confidence（信心度）**：对上述 R、I 估算的信心程度（100%=高 / 80%=中 / 50%=低）
> - **Effort（工作量）**：实现该功能需要的人月数（数值越大越费力）
>
> P0 功能 RICE 分均 ≥ 18，与优先级判断一致。当主观优先级与 RICE 分排序不一致时，在备注中说明理由。
